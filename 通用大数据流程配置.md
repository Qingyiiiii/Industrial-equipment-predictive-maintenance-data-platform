# 通用大数据流程配置

本文档只记录当前三节点虚拟机集群的通用大数据平台安装、配置、启动依赖、轻量验收和虚拟机配置排错。业务数据集实施不写在本文档中；MetroPT-3 项目已经拆分到 `MetroPT-3虚拟机测试执行清单.md`，后续其他业务项目也按独立实施文档管理。

当前目标是维护一套长期可复用的平台底座：Hadoop、Hive、Spark、Kafka、Redis、Flink、Iceberg、Doris、Trino、SeaTunnel、Flink CDC、DolphinScheduler、Prometheus 和 Grafana。本文档中的数据库、topic 和表只用于通用 demo 或组件验收，不承载具体业务项目。

---

## 0. 总体架构

### 0.1 平台分层

```text
数据源
  ├─ CSV/Parquet/JSON 文件
  ├─ Kafka 事件流
  ├─ MySQL/PostgreSQL CDC
  └─ 后续金融/电商/视频平台数据集

接入层
  ├─ Kafka KRaft
  ├─ SeaTunnel / DataX
  └─ Flink CDC

计算层
  ├─ Spark：离线批处理、ETL、特征加工
  └─ Flink：实时处理、事件时间、窗口、状态、CDC

湖仓层
  ├─ HDFS：学习环境主存储
  ├─ Hive Metastore 3.1.3：Iceberg catalog 和兼容 Hive 表
  └─ Iceberg：后续新增结果表的主表格式

查询服务层
  ├─ Doris：中国求职优先，实时分析和报表服务
  ├─ Trino：国际通用，联邦查询和湖仓查询
  └─ HiveServer2：历史兼容，不作为长期主查询入口

编排与观测
  ├─ DolphinScheduler：中国主流调度
  ├─ Airflow/dbt：国际方向后续扩展
  └─ Prometheus + Grafana：监控与可观测
```

### 0.2 固定网络

你的 VMware 网络配置：

| VMnet | 类型 | 网段 | 用途 |
|---|---|---|---|
| `VMnet8` | NAT | `192.168.159.0/24` | 三台 VM 联网、系统更新、下载依赖 |
| `VMnet10` | Host-only | `192.168.88.0/24` | 集群内固定通信、宿主机访问服务 |

三台 VM 都使用**双网卡**：

| 网卡 | VMware 设置 | Linux 作用 |
|---|---|---|
| 网卡 1 | NAT / `VMnet8` | DHCP，默认路由，访问外网 |
| 网卡 2 | Custom / `VMnet10` | 静态 IP，集群通信 |

集群固定 IP：

| 主机名 | Host-only IP |
|---|---|
| `hadoop1` | `192.168.88.101` |
| `hadoop2` | `192.168.88.102` |
| `hadoop3` | `192.168.88.103` |

所有大数据服务都绑定或暴露到 `192.168.88.101-103`，不要依赖 VMnet8 的 NAT 地址。

### 0.3 当前虚拟机资源

当前三节点虚拟机集群配置如下：

| 节点 | vCPU | 内存 | 建议磁盘 | 主要角色 |
|---|---:|---:|---:|---|
| `hadoop1` | 4 | 12GB | 140GB+ | NameNode、ResourceManager、PostgreSQL、Hive、Kafka、Redis、Doris FE、Trino coordinator、DolphinScheduler API/Master/Alert、Prometheus/Grafana |
| `hadoop2` | 2 | 7GB | 120GB+ | DataNode、NodeManager、Kafka、Flink TaskManager、Doris BE、Trino worker、DolphinScheduler worker |
| `hadoop3` | 2 | 7GB | 120GB+ | DataNode、NodeManager、Kafka、Flink TaskManager、Doris BE、Trino worker、DolphinScheduler worker |

GPU 不作为本平台依赖。后续视频分析或深度学习项目需要 GPU 时，再单独规划 CUDA/PyTorch。

### 0.4 运行模式

虚拟机重启后不要一次性拉起所有扩展组件。先按 `MetroPT-3虚拟机测试执行清单.md` 启动基础模式，确认 HDFS/YARN、PostgreSQL、Hive、Kafka、Redis、Flink 正常后，再按需启动扩展模式。

| 模式 | 常驻组件 | 按需启动 |
|---|---|---|
| 基础模式 | HDFS、YARN、PostgreSQL、Hive Metastore、HiveServer2、Kafka、Redis、Flink | Spark 小任务、Iceberg demo |
| 扩展模式 | 基础模式 + Doris、Trino、DolphinScheduler、Prometheus/Grafana | SeaTunnel/Flink CDC 模板验证 |
| 项目实施模式 | 先通过基础/扩展验收，再进入独立项目文档 | MetroPT-3 或其他业务数据集任务 |

### 3.3 双网卡配置

查看设备：

```bash
nmcli device status
nmcli con show
ip addr
ip route
```

约定：

- NAT 网卡保持 DHCP。
- Host-only 网卡设置静态 IP，不设置默认网关。

以 `hadoop1` 为例，当前 NAT 网卡设备名是 `ens160`，Host-only/Custom 网卡设备名是 `ens224`。实际名称以 `nmcli device status` 为准。

如果 `nmcli con mod "$HOST_CON"` 报 `未知的连接 ""`，说明变量没有取到连接名，或者 `ens224` 还没有 NetworkManager 连接配置。直接使用下面这种固定 profile 名的方式更稳。

```bash
nmcli con show nat-ens160 >/dev/null 2>&1 || \
  nmcli con add type ethernet ifname ens160 con-name nat-ens160

nmcli con show hostonly-ens224 >/dev/null 2>&1 || \
  nmcli con add type ethernet ifname ens224 con-name hostonly-ens224

nmcli con mod nat-ens160 \
  ipv4.method auto \
  ipv6.method disabled \
  connection.autoconnect yes

nmcli con mod hostonly-ens224 \
  ipv4.addresses 192.168.88.101/24 \
  ipv4.method manual \
  ipv4.gateway "" \
  ipv4.dns "" \
  ipv4.never-default yes \
  ipv6.method disabled \
  connection.autoconnect yes

nmcli con up nat-ens160
nmcli con up hostonly-ens224
```

`hadoop2` 改为 `192.168.88.102/24`，`hadoop3` 改为 `192.168.88.103/24`。

验证：

```bash
ip addr
ip route
ping -c 2 192.168.88.101
ping -c 2 223.5.5.5
curl -I https://www.apache.org
```

`PATH` 中必须让 `/usr/bin` 和 `$PYTHON3_HOME/bin` 排在 `$JAVA_HOME/bin` 前面。部分 JDK 软链接目录里会出现 `python` 或 `python3` 链接，如果 Java 目录排在前面，项目分析依赖会被安装或检查到错误入口。

`common` 用户再追加一个项目级简化入口，确保交互 shell 中 `python/pip` 固定使用系统 Python 3.9：

```bash
cat >> ~/.bashrc <<'EOF'

# ---- MetroPT project simplified Python/Java env ----
export JAVA_HOME=/export/server/jdk17
export PATH=/usr/local/bin:/usr/bin:/bin:$HOME/.local/bin:$JAVA_HOME/bin:$PATH

alias python='/usr/bin/python3'
alias python3='/usr/bin/python3'
alias pip='/usr/bin/python3 -m pip'
alias pip3='/usr/bin/python3 -m pip'
# ---- end MetroPT project env ----
EOF

source ~/.bashrc
```

验证：

```bash
type -a python python3
python - <<'PY'
import sys
print(sys.executable)
PY
java -version
/export/server/jdk8/bin/java -version
hadoop version
python --version
python -m pip --version
```

验收口径：

- `type -a python` 第一条应为 `alias python='/usr/bin/python3'`，或 `sys.executable` 输出 `/usr/bin/python3`。
- `python --version` 应为 Rocky 9 系统 Python 3.9。
- `java -version` 使用 JDK 17；Hive 服务端单独用 `/export/server/jdk8/bin/java`。

## 11. Iceberg 基础验收

这里创建一个平台验收用 Iceberg demo 库，不绑定任何业务域：

```bash
spark-sql <<'SQL'
CREATE DATABASE IF NOT EXISTS lakehouse.platform_demo_iceberg;

CREATE TABLE IF NOT EXISTS lakehouse.platform_demo_iceberg.demo_orders (
  id BIGINT,
  payload STRING
) USING iceberg;

INSERT INTO lakehouse.platform_demo_iceberg.demo_orders VALUES
  (1, 'from_spark_1'),
  (2, 'from_spark_2');

SELECT * FROM lakehouse.platform_demo_iceberg.demo_orders ORDER BY id;
SQL
```

Flink 写入：

```bash
/export/server/flink/bin/sql-client.sh embedded <<'SQL'
SET 'execution.runtime-mode' = 'batch';
SET 'parallelism.default' = '1';
SET 'table.dml-sync' = 'true';

CREATE CATALOG lakehouse WITH (
  'type'='iceberg',
  'catalog-type'='hive',
  'uri'='thrift://hadoop1:9083',
  'warehouse'='hdfs:///warehouse/iceberg'
);

USE CATALOG lakehouse;
USE platform_demo_iceberg;
INSERT INTO demo_orders VALUES (3, 'from_flink_3');
QUIT;
SQL
```

`INSERT` 只输出 `SQL update statement has been successfully submitted to the cluster` 和 `Job ID` 时，只代表 Flink 已经接收任务，不代表 Iceberg commit 已经完成。这里设置 `table.dml-sync=true`，让 SQL Client 等待 DML 完成后再退出。最终是否写入成功，用 Spark 交叉查询验证，不要把 `SELECT` 和 `QUIT` 一起放进 Flink SQL Client heredoc；SQL Client 的结果表是交互界面，`QUIT` 可能被当作界面按键和残留 SQL 解析，产生 `Non-query expression encountered in illegal context`。

### 13.0 扩展组件通用准备

先在 `hadoop1` 检查三台机器连通性和目录权限：

```bash

```

安装 Doris 前，三台机器都要把 `common` 用户的文件句柄数调高。Doris BE 启动脚本要求 `nofile` 大于 `60000`，建议直接设为 `655350`：

```bash

common hard nofile 655350

common hard nproc 655350

```

`limits.d` 对新登录会话生效。执行完后重新登录三台机器，或者重新打开当前终端，再确认：

```bash
for h in hadoop1 hadoop2 hadoop3; do
  ssh common@$h "ulimit -n"
done
```

如果仍然不是 `655350`，先用 `ssh common@hadoopX` 新开一个登录会话再检查。临时启动也可以在当前 shell 里执行 `ulimit -n 655350`，但前提是 hard limit 已经被提高。

本章所有 `/export/server/doris`、`/export/server/trino`、`/export/server/seatunnel` 这类路径都是软链接。同步到其他节点时不要直接同步软链接本身，优先同步真实版本目录，然后在远端重新创建软链接。

错误示例：

```bash
rsync -av /export/server/doris common@hadoop2:/export/server/doris
```

这条命令容易只复制软链接文件，或者在远端生成不符合预期的 `/export/server/doris/doris`、空链接、断链。正确模式是：

```bash
REAL_DIR=$(readlink -f /export/server/doris)
REAL_NAME=$(basename "$REAL_DIR")

rsync -av "$REAL_DIR/" common@hadoop2:/export/server/"$REAL_NAME"/
ssh common@hadoop2 "ln -sfn /export/server/$REAL_NAME /export/server/doris"
```

每次同步后都用下面命令验收：

```bash
for h in hadoop1 hadoop2 hadoop3; do
  ssh common@$h "readlink -f /export/server/doris 2>/dev/null || true; ls -ld /export/server/doris /export/server/apache-doris-* 2>/dev/null || true"
done
```

Doris BE 是本地进程，启动时需要动态加载 JDK 里的 `libjvm.so`。如果系统动态链接器找不到它，BE 会启动后立刻退出，并在 `be.out` 中报 `libjvm.so: cannot open shared object file`。三台都配置 JDK 17 的动态库路径：

```bash

  ssh common@$h '
    JDK_LIB_SERVER=$(find -L /export/server/jdk17 -path "*/lib/server/libjvm.so" -print -quit)
    if [ -z "$JDK_LIB_SERVER" ]; then
      echo "libjvm_not_found"
      exit 1
    fi
    JDK_LIB_DIR=$(dirname "$JDK_LIB_SERVER")
    echo "$JDK_LIB_DIR"
    echo "$JDK_LIB_DIR" | sudo tee /etc/ld.so.conf.d/jdk17-libjvm.conf >/dev/null
    sudo ldconfig
    ldconfig -p | grep libjvm || true
  '
done
```

## 17. 常见问题

### 17.1 NAT 能上网但集群不通

检查 VMnet10 网卡是否有 `192.168.88.x`：

```bash

nmcli con show
```

Host-only 网卡不要设置默认网关：

```bash

nmcli con up "$HOST_CON"
```

### 17.2 仍然报 `Invalid method name: get_table`

说明你还在使用 Hive 4.x Metastore 或复用了 Hive 4 初始化过的 metastore schema。

必须确认：

```bash
readlink -f /export/server/hive
grep ConnectionURL /export/server/hive/conf/hive-site.xml
grep spark.sql.catalog.lakehouse /export/server/spark/conf/spark-defaults.conf
```

要求：

- `/export/server/hive` 指向 `apache-hive-3.1.3-bin`
- PostgreSQL 库是 `metastore_hive313`
- Spark/Flink catalog 是 `thrift://hadoop1:9083`

### 17.3 虚拟机内存不足或 `hadoop1` 内存占满

优先停掉扩展组件，保留基础平台：

```bash
/export/server/flink/bin/stop-cluster.sh
for h in hadoop1 hadoop2 hadoop3; do
  ssh common@$h "/export/server/trino/bin/launcher stop 2>/dev/null || true"
done
cd /export/server/dolphinscheduler
bash bin/dolphinscheduler-daemon.sh stop api-server 2>/dev/null || true
bash bin/dolphinscheduler-daemon.sh stop master-server 2>/dev/null || true
bash bin/dolphinscheduler-daemon.sh stop alert-server 2>/dev/null || true
pkill -f '/export/server/prometheus/prometheus' 2>/dev/null || true
pkill -f 'grafana-server' 2>/dev/null || true
```

当前 VM 内存为 `hadoop1=12GB`、`hadoop2=7GB`、`hadoop3=7GB`。如果只是防止瞬时内存压力，可以在每台 VM 增加 2GB swap 作为临时缓冲；Doris BE 启动前仍必须 `swapoff -a`。

### 17.4 Trino 使用源码包、Java 版本或 worker 未加入导致启动失败

Trino 481 必须使用 Java 25：

```bash

$JAVA_HOME/bin/java -version
/export/server/trino/bin/launcher start
```

不要用 JDK 17 启动 Trino 481。

如果启动时报：

```text

```

先检查目录形态：

```bash
readlink -f /export/server/trino
ls -ld /export/server/trino /export/server/trino-*
ls -l /export/server/trino | head
test -x /export/server/trino/bin/launcher && echo trino_server_ok || echo trino_server_missing
```

如果能看到 `pom.xml`、`core/`、`testing/`，这是 Trino 源码包，不是 server 包。删除或保留源码目录都可以，但 `/export/server/trino` 软链接必须指向解压后的 `trino-server-481`：

```bash

ln -sfn /export/server/trino-server-481 /export/server/trino
mkdir -p /export/server/trino/etc/catalog /export/data/trino
test -x /export/server/trino/bin/launcher && echo trino_server_ok
```

如果三台 `launcher status` 都是 `Running`，但执行 `SELECT * FROM system.runtime.nodes` 报：

```text
Insufficient active worker nodes. Waited 5.00m for at least 1 workers, but only 0 workers are active
```

说明 coordinator 已启动，但 worker 没有加入。先检查三台配置：

```bash
for h in hadoop1 hadoop2 hadoop3; do

  ssh common@$h '
    echo "--- node.properties ---"
    cat /export/server/trino/etc/node.properties
    echo "--- config.properties ---"
    cat /export/server/trino/etc/config.properties

```

`hadoop1` 应为 `coordinator=true`，`hadoop2`、`hadoop3` 应为 `coordinator=false`，三台 `node.id` 必须分别是 `hadoop1/hadoop2/hadoop3` 或其他唯一值。

### 17.6 Doris 同步后 worker 看不到目录

```bash
for h in hadoop1 hadoop2 hadoop3; do
```

### 17.8 `Can't connect to MySQL server on '192.168.88.101'`

这个错误表示客户端已经存在，但 Doris FE 的 MySQL 协议端口 `9030` 没连上。常见原因是 FE 没启动、FE 启动失败、`9030` 没监听、IP 不对、`priority_networks` 不匹配，或者防火墙拦截。

先在 `hadoop1` 检查 FE 进程和端口：

```bash
jps -l | egrep 'DorisFE|PaloFe|Frontend|org.apache.doris' || true
ss -lntp | egrep '18030|9010|9020|9030' || true
```

如果看不到 `9030`，先看 FE 日志：

```bash
tail -n 100 /export/server/doris/fe/log/fe.out
tail -n 100 /export/server/doris/fe/log/fe.log
```

确认机器 IP 和 Doris 网络配置：

```bash
hostname -I
ip -4 addr | grep 192.168.88.101
grep priority_networks /export/server/doris/fe/conf/fe.conf
grep -n 'query_port\|http_port\|rpc_port\|edit_log_port\|meta_dir' /export/server/doris/fe/conf/fe.conf
```

`priority_networks` 必须覆盖 `hadoop1` 的真实内网 IP，例如：

```properties
priority_networks=192.168.88.0/24
```

如果 FE 没启动，重新启动：

```bash
/export/server/doris/fe/bin/stop_fe.sh --daemon 2>/dev/null || true
/export/server/doris/fe/bin/start_fe.sh --daemon
sleep 15
ss -lntp | egrep '18030|9010|9020|9030' || true
```

再分别测试本机回环地址和内网地址：

```bash
mysql -h 127.0.0.1 -P 9030 -uroot -e "SHOW FRONTENDS;"
mysql -h 192.168.88.101 -P 9030 -uroot -e "SHOW FRONTENDS;"
```

如果 `127.0.0.1` 能连、`192.168.88.101` 不能连，重点检查 FE 绑定 IP、防火墙和 `priority_networks`。学习环境可以临时关闭防火墙验证：

```bash
sudo systemctl status firewalld
sudo systemctl stop firewalld
mysql -h 192.168.88.101 -P 9030 -uroot -e "SHOW FRONTENDS;"
```

如果停防火墙后能连，再按端口放行，不要长期裸停：

```bash
sudo firewall-cmd --permanent --add-port=18030/tcp
sudo firewall-cmd --permanent --add-port=9010/tcp
sudo firewall-cmd --permanent --add-port=9020/tcp
sudo firewall-cmd --permanent --add-port=9030/tcp
sudo firewall-cmd --reload
```

### 17.9 Doris FE Web 端口与 YARN `8030` 冲突

如果启动 FE 后只看到 `8030`，但看不到 `18030/9010/9020/9030`，先不要继续执行 `ALTER SYSTEM ADD BACKEND`。`8030` 是 YARN ResourceManager 的默认端口之一；如果 YARN 已经启动，占用 `8030` 的 Java 进程大概率不是 Doris FE。

先确认占用 `8030` 的 Java 进程到底是什么：

```bash
ps -fp 181222
tr '\0' ' ' < /proc/181222/cmdline
readlink -f /proc/181222/cwd
```

把 `181222` 替换成 `ss -lntp` 里看到的实际 pid。如果 cmdline 里出现 `org.apache.hadoop.yarn.server.resourcemanager.ResourceManager`，这是 YARN ResourceManager，不要 kill。Doris FE 的 `http_port` 应改为本文档使用的 `18030`。

如果 cwd 或命令行显示是 `/export/server/doris/fe`，但只监听 `8030`、不监听 `9010/9020/9030`，才按 Doris FE 半启动残留处理。

如果确认是 Doris FE，立刻看日志，不要只重复 `start_fe.sh`：

```bash
ls -lh /export/server/doris/fe/log
tail -n 200 /export/server/doris/fe/log/fe.out
tail -n 200 /export/server/doris/fe/log/fe.log
grep -RniE "error|exception|fatal|failed|bind|9030|query_port|priority_networks|meta_dir" /export/server/doris/fe/log | tail -n 80
```

同时检查 FE 配置是否被多次追加导致混乱：

```bash
grep -nE '^(JAVA_HOME|meta_dir|priority_networks|http_port|rpc_port|query_port|edit_log_port)=' /export/server/doris/fe/conf/fe.conf
ls -ld /export/data/doris /export/data/doris/fe
```

如果发现同一个配置项重复很多次，先备份并整理 `fe.conf`。不要把文件覆盖成只有端口和目录的最小配置，否则会删除 Doris 自带的 `JAVA_OPTS_FOR_JDK_17` 等启动参数，导致 FE 无法启动。推荐先恢复最近一次备份，再只修改核心配置项：

```bash
FE_CONF=/export/server/doris/fe/conf/fe.conf
LATEST_FE_BAK=$(ls -t /export/server/doris/fe/conf/fe.conf.bak.* 2>/dev/null | head -n 1)
if [ -n "$LATEST_FE_BAK" ]; then
  cp "$LATEST_FE_BAK" "$FE_CONF"
else
  cp "$FE_CONF" "$FE_CONF.bak.$(date +%Y%m%d_%H%M%S)"
fi

set_single_fe_conf() {
  key="$1"
  value="$2"
  sed -ri "/^${key}=/d" "$FE_CONF"
  printf '%s=%s\n' "$key" "$value" >> "$FE_CONF"
}

set_single_fe_conf JAVA_HOME /export/server/jdk17
set_single_fe_conf meta_dir /export/data/doris/fe
set_single_fe_conf priority_networks 192.168.88.0/24
set_single_fe_conf http_port 18030
set_single_fe_conf rpc_port 9020
set_single_fe_conf query_port 9030
set_single_fe_conf edit_log_port 9010

mkdir -p /export/data/doris/fe
chown -R common:common /export/data/doris/fe /export/server/doris/fe/conf/fe.conf
grep -nE '^(JAVA_HOME|JAVA_OPTS_FOR_JDK_17|meta_dir|priority_networks|http_port|rpc_port|query_port|edit_log_port)=' "$FE_CONF"
```

如果 `stop_fe.sh` 已经提示 `fe.pid does not exist`，但 `ss` 又显示有 Java 进程占用 Doris FE 的端口，这属于 FE pid 文件丢失或半启动残留。确认该 pid 是 Doris FE 后，先温和停止：

```bash
kill 181222
sleep 10
ss -lntp | egrep '18030|9010|9020|9030' || true
```

如果 30 秒后进程仍存在，确认仍是 Doris FE 后再强制停止：

```bash
kill -9 181222
```

重新启动并等待更久：

```bash
/export/server/doris/fe/bin/start_fe.sh --daemon
sleep 60

jps -l | grep -i doris || true
ss -lntp | egrep '18030|9010|9020|9030' || true
tail -n 80 /export/server/doris/fe/log/fe.out
```

只有看到 `9030` 监听后，再执行：

```bash
mysql -h 192.168.88.101 -P 9030 -uroot -e "SHOW FRONTENDS;"
```

### 17.11 Doris BE Web 端口与 YARN `8040` 冲突

如果三台机器都能看到 `/export/server/doris/be/bin/start_be.sh`，但 `ss -lntp` 只看到 `8040`：

```text
LISTEN 0 256 0.0.0.0:8040 0.0.0.0:* users:(("java",pid=...,fd=...))
```

并且 `/proc/<pid>/cmdline` 显示：

```text
org.apache.hadoop.yarn.server.nodemanager.NodeManager
```

这不是 Doris 残留进程，而是 YARN NodeManager 占用了 `8040`。不要 kill 这个进程，除非你明确要停止 YARN。Doris BE 的 Web 端口需要避开 `8040`，本文档统一改用 `18040`；FE 连接 BE 仍然使用 `9050` 心跳端口。

如果 `start_be.sh --daemon` 输出：

```text
Disable swap memory before starting be
```

说明 BE 启动脚本检测到系统开启了 swap，并拒绝继续启动。先不要继续重复启动，先在三台机器关闭 swap。

如果 `start_be.sh --daemon` 输出：

```text
Set max number of open file descriptors to a value greater than 60000.
```

说明 `common` 用户当前会话的 `ulimit -n` 太低，BE 也会拒绝启动。三台机器执行：

```bash
for h in hadoop1 hadoop2 hadoop3; do
  ssh common@$h "sudo tee /etc/security/limits.d/99-doris.conf >/dev/null <<'EOF'
common soft nofile 655350
common hard nofile 655350
common soft nproc 655350
common hard nproc 655350
EOF"
done
```

重新登录后确认：

```bash
for h in hadoop1 hadoop2 hadoop3; do
  ssh common@$h "ulimit -n"
done
```

如果确认输出仍低于 `60000`，当前 SSH 会话没有加载新 limits。重新打开终端，或先单台登录 `ssh common@hadoop1` 后执行 `ulimit -n` 验证。验证通过后再启动 BE。

第一步先确认占用 `8040` 的进程是不是 YARN NodeManager：

```bash
for h in hadoop1 hadoop2 hadoop3; do
  echo "===== $h ====="
  ssh common@$h '
    BE_PID=$(ss -lntp | sed -n "s/.*:8040 .*pid=\([0-9][0-9]*\).*/\1/p" | head -n 1)
    echo "BE_PID=${BE_PID:-none}"
    if [ -n "$BE_PID" ]; then
      ps -fp "$BE_PID"
      tr "\0" " " < /proc/"$BE_PID"/cmdline
      echo
      readlink -f /proc/"$BE_PID"/cwd
    fi
  '
done
```

如果 cwd 或命令行显示是 YARN NodeManager，不要释放 `8040`，直接把 Doris BE 改成 `18040`。如果确认是 Doris BE 残留，再考虑停止残留进程。

检查并关闭 swap：

```bash
for h in hadoop1 hadoop2 hadoop3; do
  echo "===== $h ====="
  ssh common@$h "free -h; swapon --show || true"
done

for h in hadoop1 hadoop2 hadoop3; do
  ssh common@$h "sudo swapoff -a"
done

for h in hadoop1 hadoop2 hadoop3; do
  echo "===== $h ====="
  ssh common@$h "swapon --show || true"
done
```

确认 `swapon --show` 没有输出后，再考虑永久关闭。永久关闭前先备份 `/etc/fstab`：

```bash
for h in hadoop1 hadoop2 hadoop3; do
  ssh common@$h "sudo cp /etc/fstab /etc/fstab.bak.$(date +%Y%m%d_%H%M%S) && sudo sed -ri '/[[:space:]]swap[[:space:]]/ s/^/#/' /etc/fstab"
done
```

如果 `8040` 的进程名是 `java`，不要直接当作 Doris BE 成功。正常 BE 应该能看到 `doris_be` 或类似 BE 进程，并监听 `18040/9050/9060/8060`。先用上面的 `cmdline/cwd` 判断这个 Java 进程是什么；如果是 NodeManager，保留 YARN 进程并修改 Doris BE 端口。

```bash
for h in hadoop1 hadoop2 hadoop3; do
  echo "===== $h ====="
  ssh common@$h "
    grep -nE '^(JAVA_HOME|storage_root_path|priority_networks|be_port|webserver_port|heartbeat_service_port|brpc_port)=' /export/server/doris/be/conf/be.conf
    ls -ld /export/data/doris /export/data/doris/be
    ls -lh /export/server/doris/be/log || true
    tail -n 120 /export/server/doris/be/log/be.out 2>/dev/null || true
    tail -n 120 /export/server/doris/be/log/be.INFO 2>/dev/null || true
    tail -n 120 /export/server/doris/be/log/be.WARNING 2>/dev/null || true
    tail -n 120 /export/server/doris/be/log/be.ERROR 2>/dev/null || true
    grep -RniE 'error|exception|fatal|failed|storage_root_path|priority_networks|heartbeat|brpc|18040|9050|9060|8060|8040' /export/server/doris/be/log 2>/dev/null | tail -n 80
  "
done
```

如果 `be.conf` 是多次 `cat >>` 追加出来的，可能存在重复配置。先备份并重写一份干净的 BE 配置，再同步到三台：

```bash
cat > /tmp/doris_be.conf.clean <<'EOF'
JAVA_HOME=/export/server/jdk17
storage_root_path=/export/data/doris/be
priority_networks=192.168.88.0/24
be_port=9060
webserver_port=18040
heartbeat_service_port=9050
brpc_port=8060
EOF

for h in hadoop1 hadoop2 hadoop3; do
  ssh common@$h "cp /export/server/doris/be/conf/be.conf /export/server/doris/be/conf/be.conf.bak.$(date +%Y%m%d_%H%M%S) 2>/dev/null || true"
  scp /tmp/doris_be.conf.clean common@$h:/export/server/doris/be/conf/be.conf
  ssh common@$h "mkdir -p /export/data/doris/be && chown -R common:common /export/data/doris/be /export/server/doris/be/conf/be.conf"
done
```

停止半启动 BE。先用脚本停；如果脚本无法停，再只 kill 明确属于 Doris BE 的残留 pid。不要 kill YARN NodeManager：

```bash
for h in hadoop1 hadoop2 hadoop3; do
  echo "===== $h ====="
  ssh common@$h '
    /export/server/doris/be/bin/stop_be.sh --daemon 2>/dev/null || true
    sleep 5
    BE_PID=$(ps -ef | awk "/doris_be|palo_be/ && !/awk/ {print \$2; exit}")
    if [ -n "$BE_PID" ]; then
      echo "kill half-started Doris BE pid=$BE_PID"
      kill "$BE_PID"
    fi
  '
done
```

确认端口释放后重新启动：

```bash
for h in hadoop1 hadoop2 hadoop3; do
  ssh common@$h "ss -lntp | egrep '18040|9050|9060|8060' || true"
done

for h in hadoop1 hadoop2 hadoop3; do
  ssh common@$h "/export/server/doris/be/bin/start_be.sh --daemon"
done

sleep 30

for h in hadoop1 hadoop2 hadoop3; do
  echo "===== $h ====="

done

```

如果仍然没有 `18040/9050/9060/8060`，不要继续重复启动，把三台 `be.out`、`be.WARNING`、`be.ERROR` 中最后的错误贴出来，优先按日志根因处理。

### 17.16 DolphinScheduler Worker SSH 到业务执行节点失败

判断：Worker 节点的 `~/.ssh/known_hosts` 中没有 `hadoop1` 指纹，或已有指纹与当前虚拟机不一致。DolphinScheduler Shell 任务是非交互执行，无法输入 `yes` 确认新指纹。

修正：

```bash
for h in hadoop2 hadoop3; do
```

### 17.18 Kafka 使用 JDK 8 导致 `UnsupportedClassVersionError`

现象：

```text
UnsupportedClassVersionError: org/apache/kafka/tools/TopicCommand has been compiled by a more recent version of the Java Runtime (class file version 61.0), this version of the Java Runtime only recognizes class file versions up to 52.0
```

判断：Kafka 4.x 需要 Java 17，`class file version 61.0` 对应 Java 17，`52.0` 对应 Java 8。通常是刚启动 Hive 后当前 shell 还保留 `JAVA_HOME=/export/server/jdk8`，导致 Kafka 服务端和 CLI 都用 JDK 8 启动。

修正：Kafka 启动和 Kafka CLI 前都显式切回 JDK 17。

```bash
export JAVA_HOME=/export/server/jdk17
export PATH=/usr/local/bin:/usr/bin:/bin:$JAVA_HOME/bin:$PATH
java -version

for h in hadoop1 hadoop2 hadoop3; do
  ssh common@$h "pkill -f 'kafka.Kafka' 2>/dev/null || true"
done

for h in hadoop1 hadoop2 hadoop3; do
  ssh common@$h "export JAVA_HOME=/export/server/jdk17; export PATH=/usr/local/bin:/usr/bin:/bin:\$JAVA_HOME/bin:\$PATH; mkdir -p /export/logs/kafka && nohup /export/server/kafka/bin/kafka-server-start.sh /export/server/kafka/config/kraft/server.properties > /export/logs/kafka/kafka-server.out 2>&1 &"
done

sleep 10
/export/server/kafka/bin/kafka-metadata-quorum.sh --bootstrap-server 192.168.88.101:9092 describe --status
/export/server/kafka/bin/kafka-topics.sh --bootstrap-server 192.168.88.101:9092 --list
ss -lntp | egrep '9092|9093' || true
```

如果三台日志只看到 `Connection to node ...:9093 could not be established`，并且检查时没有任何 `9092/9093` 监听，说明 Kafka 没有真正同时拉起形成 KRaft quorum。改用 `ssh -n + setsid + </dev/null` 彻底脱离 SSH 会话：

```bash
for h in hadoop1 hadoop2 hadoop3; do
  ssh -n common@$h "bash -lc 'export JAVA_HOME=/export/server/jdk17; export PATH=/usr/local/bin:/usr/bin:/bin:\$JAVA_HOME/bin:\$PATH; mkdir -p /export/logs/kafka; setsid /export/server/kafka/bin/kafka-server-start.sh /export/server/kafka/config/kraft/server.properties > /export/logs/kafka/kafka-server.out 2>&1 < /dev/null &'"
done
```

### 17.19 Trino Iceberg 能列库表但读表报 `Error processing metadata`

现象一：Trino 服务已启动，`SHOW CATALOGS` 和 `SHOW TABLES` 正常，但读取 Iceberg 表时报：

```text
Error processing metadata for table <schema>.<table>
```

现象二：为了解决读表问题，在 `iceberg.properties` 中只加入：

```properties
hive.config.resources=/export/server/hadoop/etc/hadoop/core-site.xml,/export/server/hadoop/etc/hadoop/hdfs-site.xml
```

然后 Trino 直接启动失败，`server.log` 报：

```text
Configuration property 'hive.config.resources' was not used
```

判断：当前 Iceberg 表的数据和 metadata 文件在 HDFS 上。Trino 481 的 Iceberg catalog 需要显式启用 HDFS 文件系统支持，否则能通过 Hive Metastore 看见库表，但读 Iceberg metadata/data 文件时失败。`hive.config.resources` 不能单独作为 Iceberg catalog 配置加入，必须和 `fs.hadoop.enabled=true` 一起使用。

修正：三台 Trino 都使用下面的 `/export/server/trino/etc/catalog/iceberg.properties`：

```bash
cat > /export/server/trino/etc/catalog/iceberg.properties <<'EOF'
connector.name=iceberg
iceberg.catalog.type=hive_metastore
hive.metastore.uri=thrift://hadoop1:9083
fs.hadoop.enabled=true
hive.config.resources=/export/server/hadoop/etc/hadoop/core-site.xml,/export/server/hadoop/etc/hadoop/hdfs-site.xml
EOF

for h in hadoop2 hadoop3; do
  scp /export/server/trino/etc/catalog/iceberg.properties common@$h:/export/server/trino/etc/catalog/iceberg.properties
done

for h in hadoop1 hadoop2 hadoop3; do
  ssh common@$h "export JAVA_HOME=/export/server/jdk25; /export/server/trino/bin/launcher restart"
done
```

验收：

```bash
trino --server http://hadoop1:8080 --execute "SELECT node_id,http_uri,node_version,coordinator,state FROM system.runtime.nodes"
trino --server http://hadoop1:8080 --execute "SHOW SCHEMAS FROM iceberg"
trino --server http://hadoop1:8080 --execute "SHOW TABLES FROM iceberg.<schema_name>"
trino --server http://hadoop1:8080 --execute "SELECT COUNT(*) FROM iceberg.<schema_name>.<table_name>"
```

如果 `SHOW SCHEMAS FROM lakehouse` 报：

```text
Catalog 'lakehouse' not found
```

说明 catalog 文件名不是 `lakehouse.properties`。当前平台默认文件名是 `iceberg.properties`，查询前缀应改为 `iceberg`。

### 17.20 DolphinScheduler Shell 节点保存时报 `请求参数[{0}]无效`

现象：在 DolphinScheduler 工作流定义里创建 `SHELL` 节点，脚本内容只有：

```bash
echo 123
```

点击保存工作流基本信息后仍报：

```text
请求参数[{0}]无效
```

或者创建 `SSH` 数据源时报：

```text
Cannot invoke "...DataSourceProcessor.castDatasourceParamDTO(String)" because the return value of "DataSourceUtils.getDatasourceProcessor(...)" is null
```

判断：这不是脚本内容错误，也不是远程主机名错误，而是 DolphinScheduler 后端没有加载对应插件。`SHELL` 节点依赖 task-shell 插件，`SSH` 数据源或 RemoteShell 依赖对应数据源/任务插件。插件缺失时前端报错很泛，真实原因通常在 API 日志里。

修正：

```bash
cd /export/server/dolphinscheduler

bash bin/install-plugins.sh 3.4.1

find /export/server/dolphinscheduler -type f \
  | egrep -i 'dolphinscheduler-task-shell|task-shell|datasource|ssh|remoteshell' \
  | sort

DS_REAL_DIR=$(readlink -f /export/server/dolphinscheduler)
DS_REAL_NAME=$(basename "$DS_REAL_DIR")

for h in hadoop2 hadoop3; do
  rsync -av "$DS_REAL_DIR/" common@$h:/export/server/"$DS_REAL_NAME"/
  ssh common@$h "ln -sfn /export/server/$DS_REAL_NAME /export/server/dolphinscheduler"
done

bash bin/dolphinscheduler-daemon.sh restart api-server
bash bin/dolphinscheduler-daemon.sh restart master-server
ssh common@hadoop2 "cd /export/server/dolphinscheduler && bash bin/dolphinscheduler-daemon.sh restart worker-server"
ssh common@hadoop3 "cd /export/server/dolphinscheduler && bash bin/dolphinscheduler-daemon.sh restart worker-server"
```

如果 `restart` 子命令不可用，就按 13.5 节中的 stop/start 方式重启。验收时先创建一个只有 `echo 123` 的 Shell 工作流，能保存、上线并生成任务实例后，再接入业务脚本。
