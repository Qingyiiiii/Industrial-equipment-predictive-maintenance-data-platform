## How To Read Logs

Do not start with large logs. Recommended order:

1. `summary.tsv`: check `FAIL` first, then `WARN` and `SKIP`.
2. `*_steps.tsv`: for P14, locate the exact failed step.
3. `command.tsv`: confirm parameters and mode.
4. Related `.log`: inspect the first error, return code, or SQL rc.
5. `readiness.tsv`: for P7, determine whether the issue is service availability or resource headroom.

Status meanings:

| Status | Meaning |
| --- | --- |
| `PASS` | Step passed |
| `WARN` | Risk or boundary note; not necessarily failure |
| `SKIP` | Explicitly skipped; cannot be presented as full PASS |
| `FAIL` | Failed; inspect logs and rerun after fixing |
| `PASS_WITH_WARNINGS` | Business pipeline passed, but WARN boundary exists |

## SQL Files

SQL files mainly live in:

```text
analysis/bi/p13_dashboard_materials/sql/
validation_runs/*/sql/
```

Read them by checking:

1. Query engine: `hive`, `trino`, `doris`, or analysis artifact.
2. Input object: Hive table, Trino schema, Doris table, or local TSV.
3. Output location: `sample_outputs/`, `p12_query_results.tsv`, or related `.log`.

For P12, the point is not a single attractive SQL query; the key evidence is Hive vs Trino and Hive vs Doris consistency `PASS`.

## FAQ

| Question | Answer |
| --- | --- |
| There are many `validation_runs`; which one should I read? | Sort by timestamp and prefer the prefix for the current task |
| A `.log` file is too long | Use `summary.tsv` first to locate the failed step, then read the related log |
| How should `.parquet` files be opened? | Avoid manually opening large files; use generation scripts or sample TSVs |
| Are `.png` files final dashboards? | No. They are analysis figures and BI materials |
| Which is authoritative, Chinese reports or English originals? | English originals and run directories are evidence artifacts; Chinese reports are quick-reading guides |
